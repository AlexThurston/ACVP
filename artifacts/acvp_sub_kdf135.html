1. Introduction

The Automated Crypto Validation Protocol (ACVP) defines a mechanism to automatically verify the cryptographic implementation of a software or hardware crypto module. The ACVP specification defines how a crypto module communicates with an ACVP server, including crypto capabilities negotiation, session management, authentication, vector processing and more. The ACVP specification does not define algorithm specific JSON constructs for performing the crypto validation. A series of ACVP sub-specifications define the constructs for testing individual crypto algorithms. Each sub-specification addresses a specific class of crypto algorithms. This sub-specification defines the JSON constructs for testing SP800-135 KDF algorithms using ACVP.

1.1. Requirements Language

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted in RFC 2119 [RFC2119].

2. Capabilities Registration

ACVP requires crypto modules to register their capabilities. This allows the crypto module to advertise support for specific algorithms, notifying the ACVP server which algorithms need test vectors generated for the validation process. This section describes the constructs for advertising support of SP800-135 KDF algorithms to the ACVP server.

The algorithm capabilities are advertised as JSON objects within the 'algorithms' value of the ACVP registration message. The 'algorithms' value is an array, where each array element is an individual JSON object defined in this section. The 'algorithms' value is part of the 'capability_exchange' element of the ACVP JSON registration message. See the ACVP specification for details on the registration message. Each algorithm capability advertised is a self-contained JSON object. The following JSON values are used for SP800-135 KDFalgorithm capabilities:

SP800-135 KDF Algorithm Capabilities JSON Values
JSON Value Description JSON type Valid Values Optional
algorithm The KDF to be validated. value See Section 2.1 No
aesval The TDES validation number or implemented in same implementation(SRTP). value same or value Yes
hmacval The HMAC validation number or implemented in same implementation(IKEv1, IKEv2, TLS, TPM). value same or value Yes
shaval The SHA validation number or implemented in same implementation(IKEv1, IKEv2, ANS X9.63-2001, SSH, SNMP, TLS, TPM). value same or value Yes
methods Type of method or algorithm supported array See Section 2.2 Yes
sha SHA functions supported array SHA-1, SHA-224, SHA-256, SHA-384, SHA-512 Yes
keylen Key length minimum and maximum array 0-65536 Yes
EngineID SNMP: Two engine IDs in hexidecimal, can be the same array 0-0xF Yes
plen SNMP: Password minimum and maximum lengths array alphanumeric Yes
kdr0 SRTP: Key Derivation Rate 0 array all, 1-24 Yes
kdr1 SRTP: Key Derivation Rate 1 array all, 1-24 Yes
field ANSI X9.63: Minimum and Maximum field size in array 0-65536 Yes
sharedinfo ANSI X9.63: Minimum and Maximum sharedinfo size in bits array 0-1024 Yes
initiator IKE: Minimum and maximum lengths in bytes array 0-65536 Yes
responder IKE: Minimum and maximum lengths in bytes array 0-65536 Yes
psk IKE: Pre-shared key minimum and maximum lengths in bytes array 0-65536 Yes
dhkey IKE: Three Diffie-Hellman shared secret key lengths in bytes array 0-65536 Yes
dkmlen IKE: Minimum and maximum lengths of the 7 keys derived by SKEYSEED in bytes array 0-65536 Yes

Note: Some optional values are required depending on the algorithm. For example, SHA-1 requires inbyte, inempty, inlen, and outlen. Failure to provide these values will result in the ACVP server returning an error to the ACVP client during registration.

2.1. Supported SP800-135 KDF Algorithms

The following SP800-135 KDF algorithms may be advertised by the ACVP compliant crypto module:

  • IKEv1
  • IKEv2
  • TLS
  • ANSI X9.63
  • SSH
  • SRTP
  • SNMP
  • TPM

2.2. Supported SP800-135 KDF Algorithms

The following SP800-135 KDF algorithms may be advertised by the ACVP compliant crypto module:

  • Digital Signature(IKEv1)
  • Public Key Encryption(IKEv1)
  • Pre-shared Key(IKEv1)
  • TLS 1.0/1.1(TLS)
  • TLS 1.2(TLS)
  • TDES CBC(SSH)
  • AES-128-CBC(SSH)
  • AES-192-CBC(SSH)
  • AES-256-CBC(SSH)
  • AES-128-CBC(SRTP)
  • AES-192-CBC(SRTP)
  • AES-256-CBC(SRTP)

3. Test Vectors

The ACVP server provides test vectors to the ACVP client, which are then processed and returned to the ACVP server for validation. A typical ACVP validation session would require multiple test vector sets to be downloaded and processed by the ACVP client. Each test vector set represents an individual Key Derivation Function (KDF), such as SNMP, SSH, etc. This section describes the JSON schema for a test vector set used with SP800-135 KDF algorithms.

The test vector set JSON schema is a multi-level hierarchy that contains meta data for the entire vector set as well as individual test vectors to be processed by the ACVP client. The following table describes the JSON elements at the top level of the hierarchy.

Vector Set JSON Object
JSON Value Description JSON type
acv_version Protocol version identifier value
vs_id Unique numeric identifier for the vector set value
algorithm The hash algorithm and mode used for the test vectors. See Section 2.1 for possible values. value
test_groups Array of test group JSON objects, which are defined in Section 3.1 array

3.1. Test Groups JSON Schema

The test_groups element at the top level in the test vector JSON object is an array of test groups. Test vectors are grouped into similar test cases to reduce the amount of data transmitted in the vector set. For instance, all test vectors that use the same key size would be grouped together. The Test Group JSON object contains meta data that applies to all test vectors within the group. The following table describes the SP800-135 KDF IKEv1 JSON elements of the Test Group JSON object.

Test Group JSON Object
JSON Value Description JSON type Optional
tests Array of individual test vector JSON objects, which are defined in Section 3.2 array No

The following table describes the SP800-135 KDF IKEv2 JSON elements of the Test Group JSON object.

Test Group JSON Object
JSON Value Description JSON type Optional
tests Array of individual test vector JSON objects, which are defined in Section 3.2 array No

The following table describes the SP800-135 KDF ANS X9.63 JSON elements of the Test Group JSON object.

Test Group JSON Object
JSON Value Description JSON type Optional
field Minimum and maximum field size array No
sharedinfo Minimum and maximum sharedinfo length array No
keylen Minimum and maximum key lengths, maximum less than or equal to 4096 array No
sha SHA functions supported array No
tests Array of individual test vector JSON objects, which are defined in Section 3.2 array No

The following table describes the SP800-135 KDF SSH JSON elements of the Test Group JSON object.

Test Group JSON Object
JSON Value Description JSON type Optional

3.2. Test Vectors JSON Schema

Each test group contains an array of one or more test vectors. Each test vector is a JSON object that represents a single test case to be processed by the ACVP client. The following table describes the JSON elements for each SP800-135 KDF test vector.

Test Case JSON Object
JSON Value Description JSON type Optional
tc_id Unique numeric identifier for the test case value No
len Length of the message or seed value No
msg Value of the message or seed value No
md Hash value generated value Yes

4. Test Vector Responses

After the ACVP client downloads and processes a vector set, it must send the resonse vectors back to the ACVP server. The following table describes the JSON object that represents a vector set response.

Vector Set Response JSON Object
JSON Value Description JSON type
acv_version Protocol version identifier value
vs_id Unique numeric identifier for the vector set value
test_results Array of JSON objects that represent each test vector result, which uses the same JSON schema as defined in Section 3.2 array

Appendix A. Example SP800-135 KDF Capabilities JSON Object

The following is a example JSON object advertising support for SHA-256.



            {
                "algorithm": "SHA-256",
                "inbyte": "yes",
                "inempty": "yes",
            }

The following is a example JSON object advertising support for HMAC-SHA-1.



            {
                "algorithm": "HMAC-SHA-256",
                "keysize1": [ 32, 56],
                "keysize2": [192, 256],
                "keyblock": "yes",
                "maclen": [20],
            }
